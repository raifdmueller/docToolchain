== Deployment View

docToolchain supports three execution environments and is distributed through multiple channels.
The `dtcw` wrapper abstracts environment differences so that users interact with a single CLI regardless of the underlying installation method.

=== Infrastructure Level 1: Execution Environments

[plantuml,deployment-overview,png]
----
@startuml
!include <C4/C4_Deployment>

LAYOUT_WITH_LEGEND()

Deployment_Node(workstation, "Developer Workstation", "Linux / macOS / Windows") {

    Deployment_Node(local, "Local Installation", "$HOME/.doctoolchain/") {
        Deployment_Node(jdk, "Adoptium JDK 17", "Auto-installed by dtcw") {
            Container(dtcLocal, "docToolchain", "Groovy/Gradle", "Full installation in\n$HOME/.doctoolchain/\ndocToolchain-<version>/")
        }
    }

    Deployment_Node(docker, "Docker Engine", "Container runtime") {
        Container(dtcDocker, "docToolchain Container", "doctoolchain/doctoolchain:v<version>", "Alpine Linux + JDK 17\nProject mounted as /project")
    }

    Deployment_Node(sdk, "SDKMAN", "$HOME/.sdkman/") {
        Container(dtcSdk, "docToolchain", "Groovy/Gradle", "SDK-managed installation in\n$HOME/.sdkman/candidates/\ndoctoolchain/<version>/")
    }
}

Deployment_Node(ci, "GitHub Actions", "CI/CD Platform") {
    Container(ciRunner, "CI Runner", "Ubuntu + JDK 17", "Runs .ci.sh pipeline\nCore tests + doc generation")
}

Deployment_Node(dist, "Distribution Channels", "") {
    Container(ghRelease, "GitHub Releases", "", "ZIP distribution\ndocToolchain-<version>.zip")
    Container(dockerHub, "Docker Hub", "", "Container images\ndoctoolchain/doctoolchain")
    Container(sdkmanApi, "SDKMAN API", "", "SDK package registry")
}

Rel(ciRunner, ghRelease, "Publishes release ZIP")
Rel(ciRunner, dockerHub, "Triggers Docker build")
Rel(ciRunner, sdkmanApi, "Registers new version")

Rel(dtcLocal, ghRelease, "Downloads from")
Rel(dtcDocker, dockerHub, "Pulls from")
Rel(dtcSdk, sdkmanApi, "Installs from")
@enduml
----

Motivation::
Supporting three execution environments maximizes adoption across different developer setups and CI/CD platforms. Local installation offers the fastest feedback loop. Docker provides reproducible builds without any local tool installation. SDKMAN integrates with the JVM ecosystem's standard package manager.

Quality and/or Performance Features::

[options="header",cols="1,2,2,2"]
|===
|Environment |Advantages |Limitations |Typical Use

|Local
|Fastest startup (~13s for HTML). Direct filesystem access. Full Gradle caching.
|Requires Java 17 on the host (auto-installed by dtcw).
|Daily development, documentation preview

|Docker
|No local dependencies beyond Docker. Guaranteed consistent environment across machines (QS-2).
|Slower I/O due to volume mounts. No Gradle daemon persistence between runs.
|CI/CD pipelines, team standardization

|SDKMAN
|Familiar JVM ecosystem tooling. Version switching (`sdk use doctoolchain <version>`).
|Requires SDKMAN installation. Limited to Unix-like systems.
|JVM developers, multi-version testing
|===

Mapping of Building Blocks to Infrastructure::

[options="header",cols="1,3"]
|===
|Building Block |Infrastructure Mapping

|Wrapper Layer (`dtcw`)
|Runs directly on the host shell. Detects available environments and delegates accordingly. In Docker mode, it constructs the `docker run` command with volume mounts and environment variable passthrough.

|Gradle Orchestration
|Runs inside the JVM (local or container). Uses `$HOME/.gradle/` for caching (local/SDKMAN) or a container-internal cache (Docker).

|Core Library
|Distributed as `libs/org/docToolchain/core-<version>.jar` inside the docToolchain installation directory. Loaded by Gradle at build time.

|Script Plugins
|Located in the `scripts/` directory of the installation. Applied by `build.gradle` relative to the docToolchain home directory.
|===

// Source: dtcw (environment detection and installation logic)

=== Infrastructure Level 2: CI/CD Pipeline

[plantuml,deployment-cicd,png]
----
@startuml
rectangle "GitHub Repository" as repo {
    rectangle "Push to ng branch\nor Pull Request" as trigger
}

rectangle "GitHub Actions" as gha {
    rectangle "default-build.yml" as build {
        rectangle "1. Install system packages\n(graphviz, shellcheck, pandoc)" as step1

            rectangle "b) ./gradlew dependencyUpdates" as ci2
            rectangle "c) ./gradlew core:test" as ci3
            rectangle "d) ./gradlew test" as ci4
            rectangle "e) ./dtcw local generateHTML\n./dtcw local generatePDF\n./dtcw local generateSite" as ci5

    }
}

rectangle "Release Pipeline" as release {
    rectangle "release.yml\n(manual dispatch)" as rel
    rectangle "Create GitHub Release\n+ ZIP distribution" as relGh
    rectangle "Trigger docker-image\nrepository build" as relDocker
    rectangle "Register with SDKMAN" as relSdk

trigger --> build
ci1 --> ci2
ci2 --> ci3
ci3 --> ci4
ci4 --> ci5
step3 --> step4

rel --> relGh
relGh --> relDocker
relGh --> relSdk
@enduml
----

The CI pipeline (`default-build.yml`) runs on every push to the `ng` branch and on pull requests.
It installs system dependencies, runs the full test suite, and generates documentation.

The release pipeline (`release.yml`) is triggered manually and creates a GitHub Release with a ZIP distribution, triggers Docker image builds in the `docker-image` repository, and registers the new version with SDKMAN.
