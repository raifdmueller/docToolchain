== Building Block View

This section decomposes docToolchain into its static building blocks at three levels of detail.

=== Level 1: Whitebox Overall System

[plantuml,level1-container,png]
----

Motivation::
docToolchain's four-layer architecture separates concerns cleanly: the wrapper handles platform abstraction, Gradle provides build orchestration, script plugins define features, and the core library contains reusable business logic.

Contained Building Blocks::

[options="header",cols="1,2,2"]
|===
|Building Block |Responsibility |Location

|Wrapper Layer
|Cross-platform CLI entry point. Detects execution environment (local/Docker/SDKMAN), manages Java 17 installation, invokes Gradle with correct arguments.
|`dtcw`, `dtcw.ps1`, `dtcw.bat`

|Gradle Orchestration
|Build configuration, task registration via `apply from:`, dependency resolution, plugin loading. Central coordination point for all tasks.
|`build.gradle`, `settings.gradle`, `gradle.properties`, `libs.versions.toml`

|Script Plugins
|Feature-specific Gradle task definitions. Each script is a self-contained module that registers tasks in the `docToolchain` group.
|`scripts/*.gradle` (25+ files)

|Core Library
|Business logic independent of Gradle. Provides services for Confluence publishing, Jira export, configuration management, and HTTP communication. Packaged as a Shadow JAR.
|`core/src/main/groovy/org/docToolchain/`
|===

==== Tasks

* **Purpose**: Gradle task implementations. Each task class encapsulates the logic for one docToolchain operation.
* **Key classes**: `DocToolchainTask` (abstract base), `Asciidoc2ConfluenceTask`, `ExportJiraIssuesTask`, `ExportJiraSprintChangelogTask`, `VerifyConfluenceApiAccessTask`, `WipeConfluenceSpaceTask`
* **Location**: `core/src/main/groovy/org/docToolchain/tasks/`

==== Configuration

* **Purpose**: Load and provide typed access to project configuration.
* **Key classes**: `ConfigBuilder` (bootstraps ConfigObject from Groovy DSL file), `ConfigService` (provides `getConfigProperty(path)` and `getFlatConfigSubTree(path)`)
* **Location**: `core/src/main/groovy/org/docToolchain/configuration/`

==== Confluence Subsystem

* **Purpose**: Publish HTML documentation to Confluence. Handles page creation/update, attachment upload, label management, and content hash comparison for idempotent publishing.
* **Key classes**: `ConfluenceService`, `ConfluenceClient` (abstract), `ConfluenceClientV1`, `ConfluenceClientV2`, `RestClient`
* **Location**: `core/src/main/groovy/org/docToolchain/atlassian/confluence/`

==== Jira Subsystem

* **Purpose**: Export Jira issues and sprint data to AsciiDoc or Excel format.
* **Key classes**: `JiraService`, `JiraClient` (abstract), `JiraServerClient`, `AsciiDocConverter`, `ExcelConverter`, `LegacyJiraProcessor`
* **Location**: `core/src/main/groovy/org/docToolchain/atlassian/jira/`

==== HTTP Layer

* **Purpose**: Generic HTTP client abstraction for REST API calls. Handles authentication, error responses, and proxy configuration.
* **Key classes**: `BasicRestClient`, `RequestFailedException`
* **Location**: `core/src/main/groovy/org/docToolchain/http/`

==== Transformers

* **Purpose**: Transform HTML output from AsciiDoctor into Confluence storage format. Handles code blocks, links, anchors, admonitions, and embedded images.
* **Key classes**: `HtmlTransformer`, `CodeBlockTransformer`, `LinkTransformer`
* **Location**: `core/src/main/groovy/org/docToolchain/atlassian/transformer/`

=== Level 2: Script Plugin Categories

The 25+ script plugins are organized into six functional groups:

[options="header",cols="1,2,3"]

|Category |Scripts |Purpose

|Document Generation
|`AsciiDocBasics.gradle`, `generateDeck.gradle`
|Core document generation: `generateHTML`, `generatePDF`, `generateDocbook`, `generateDeck` (Reveal.js slides). Configuration loading and AsciiDoctor setup.

|Atlassian Integration
|`publishToConfluence.gradle`, `exportJiraIssues.gradle`, `exportJiraSprintChangelog.gradle`, `exportConfluence.gradle`
|Confluence publishing and Jira data export. These scripts delegate to core module task classes.

|External Tool Export
|`exportEA.gradle`, `exportPPT.gradle`, `exportVisio.gradle`, `exportStructurizr.gradle`, `exportOpenApi.gradle`
|Extract diagrams and models from Enterprise Architect, PowerPoint, Visio, Structurizr DSL, and OpenAPI specs.

|`exportExcel.gradle`, `exportMarkdown.gradle`, `exportChangelog.gradle`, `exportContributors.gradle`, `exportMetrics.gradle`, `pandoc.gradle`
|Import data from Excel tables, convert Markdown, extract Git metadata, compute documentation metrics, and run Pandoc conversions.

|Site Generation
|`generateSite.gradle`, `copyThemes.gradle`, `generateContent.gradle`
|Generate static microsites using jBake. Handle theme management, content preparation, and jBake configuration.

|Utilities
|`collectIncludes.gradle`, `fixEncoding.gradle`, `prependFilename.gradle`, `fixGlobalReferences.gradle`, `htmlSanityCheck.gradle`, `downloadTemplate.gradle`, `customTasks.gradle`
|Supporting tasks: aggregate AsciiDoc includes, fix file encodings, validate HTML output, download arc42 templates, and user-defined custom tasks.

==== Confluence Publishing Flow

The Confluence subsystem implements a **smart publishing** pattern:

1. `ConfluenceService` parses generated HTML files using jsoup
2. For each page, it computes an MD5 hash of the content
3. It queries Confluence for existing pages in the target space
4. If a page exists and its hash matches, the update is skipped (QS-4)
5. If content changed, `HtmlTransformer` converts HTML to Confluence storage format
6. `ConfluenceClient` (V1 or V2, based on config) creates or updates the page
7. Attachments are uploaded only if their content changed
8. Labels are applied or updated

// Source: core/src/main/groovy/org/docToolchain/tasks/Asciidoc2ConfluenceTask.groovy,
// core/src/main/groovy/org/docToolchain/atlassian/confluence/ConfluenceService.groovy

==== Jira Export Flow

The Jira subsystem implements a **converter chain** pattern:

1. `JiraService` receives export configuration (JQL queries, output format, custom fields)
2. `JiraServerClient` executes JQL queries via REST API, handling pagination
3. For each export entry, the configured converter is invoked:
   - `AsciiDocConverter`: Formats issues as AsciiDoc tables with configurable columns
   - `ExcelConverter`: Creates XLSX workbooks via Apache POI with one sheet per export
4. `LegacyJiraProcessor` handles backward-compatible configuration format from older docToolchain versions

// core/src/main/groovy/org/docToolchain/atlassian/jira/clients/JiraServerClient.groovy
