== Runtime View

The following scenarios illustrate how docToolchain's building blocks interact at runtime for the most architecturally significant use cases.

=== Scenario 1: Generate HTML Documentation

The most common user workflow: converting AsciiDoc sources to HTML.

[plantuml,runtime-generatehtml,png]
----
@startuml
actor "Author" as user
participant "dtcw" as dtcw
participant "Gradle" as gradle
participant "AsciiDocBasics.gradle" as basics
participant "ConfigBuilder" as cb
participant "AsciiDoctor JVM" as asciidoc

user -> dtcw : ./dtcw local generateHTML
activate dtcw
dtcw -> dtcw : Detect environment\n(local/docker/sdk)
dtcw -> dtcw : Verify Java 17
dtcw -> gradle : ./gradlew generateHTML\n-PmainConfigFile=...
activate gradle
gradle -> basics : apply from: 'scripts/AsciiDocBasics.gradle'
activate basics
basics -> cb : ConfigBuilder.build()
activate cb
cb -> cb : Parse docToolchainConfig.groovy\nvia ConfigSlurper
cb --> basics : ConfigObject
deactivate cb
basics -> asciidoc : asciidoctor task\n(inputFiles, outputPath, imageDirs)
activate asciidoc
asciidoc -> asciidoc : Process AsciiDoc\nRender PlantUML diagrams\nApply PDF theme
asciidoc --> basics : build/html5/*.html
deactivate asciidoc
basics --> gradle : Task complete
deactivate basics
gradle --> dtcw : Exit code 0
deactivate gradle
dtcw --> user : HTML in build/html5/
deactivate dtcw
@enduml
----

**Notable aspects:**

* `dtcw` never calls AsciiDoctor directly -- all tool invocation is delegated through Gradle's task dependency graph.
* Configuration is loaded once by `ConfigBuilder` and shared across all tasks in the build.
* PlantUML diagrams are rendered inline by the `asciidoctorj-diagram` extension during AsciiDoc processing.

// Source: dtcw, scripts/AsciiDocBasics.gradle, core/src/main/groovy/org/docToolchain/configuration/ConfigBuilder.groovy

=== Scenario 2: Publish to Confluence

The publishing flow demonstrates the Confluence subsystem's smart update mechanism (QS-4).

[plantuml,runtime-confluence,png]
----
@startuml
actor "Author" as user
participant "dtcw" as dtcw
participant "Gradle" as gradle
participant "Asciidoc2Confluence\nTask" as task
participant "ConfluenceService" as svc
participant "HtmlTransformer" as trans
participant "ConfluenceClient\n(V1 or V2)" as client
participant "Confluence\nREST API" as api

user -> dtcw : ./dtcw local publishToConfluence
dtcw -> gradle : ./gradlew publishToConfluence
gradle -> gradle : Resolve dependency:\ngenerateHTML first
gradle -> task : execute()
activate task

task -> task : Parse HTML files\nfrom build/html5/
task -> task : Extract page hierarchy\nfrom section structure

loop for each page
    task -> task : Compute MD5 hash\nof page content
    task -> svc : Check if page exists
    activate svc
    svc -> client : Fetch existing pages
    client -> api : GET /pages?spaceKey=...
    api --> client : Page list with hashes
    client --> svc : Existing page map
    deactivate svc

    alt Content unchanged (hash match)
        task -> task : Skip update
    else Content changed or new page
        task -> trans : transformToConfluenceFormat(html)
        activate trans
        trans -> trans : Rewrite links and anchors
        trans -> trans : Convert code blocks\nto Confluence macros
        trans -> trans : Convert admonitions\nto info/warning panels
        trans --> task : Confluence storage format
        deactivate trans

        task -> client : Create or update page
        client -> api : POST/PUT /pages
        api --> client : Page ID

        task -> client : Upload changed attachments
        client -> api : POST /attachments

        task -> client : Apply labels
        client -> api : POST /labels
    end
end

task --> gradle : Complete
gradle --> dtcw : Exit code 0
dtcw --> user : Pages published
deactivate task
@enduml
----

**Notable aspects:**

* The task automatically depends on `generateHTML` -- Gradle resolves this before publishing.
* MD5 hash comparison prevents unnecessary Confluence page version creation.
* `HtmlTransformer` chains through `CodeBlockTransformer` and `LinkTransformer` for format conversion.
* Client selection (V1 vs V2) is determined by the `confluence.useV1Api` configuration flag.

// Source: core/src/main/groovy/org/docToolchain/tasks/Asciidoc2ConfluenceTask.groovy,
// core/src/main/groovy/org/docToolchain/atlassian/confluence/ConfluenceService.groovy

=== Scenario 3: Export Jira Issues

Demonstrates the converter chain pattern for exporting Jira data.

[plantuml,runtime-jira,png]
----
@startuml
actor "Author" as user
participant "Gradle" as gradle
participant "ExportJiraIssues\nTask" as task
participant "JiraService" as svc
participant "JiraServerClient" as client
participant "Jira REST API" as api
participant "AsciiDocConverter" as adoc
participant "ExcelConverter" as excel

user -> gradle : ./dtcw local exportJiraIssues
gradle -> task : execute()
activate task

task -> svc : exportJira(config.jira)
activate svc

loop for each export entry in config
    svc -> client : getIssuesByJql(jql, fields)
    activate client
    client -> api : GET /rest/api/3/search?jql=...
    api --> client : JSON issue list (paginated)
    client --> svc : List<Issue>
    deactivate client

    alt format == "asciidoc"
        svc -> adoc : convert(issues, customFields)
        adoc --> svc : AsciiDoc table string
        svc -> svc : Write to build/jira/<filename>.adoc
    else format == "excel"
        svc -> excel : convert(issues, customFields)
        excel --> svc : XLSX workbook
        svc -> svc : Write to build/jira/<filename>.xlsx
    end
end

svc --> task : Export complete
deactivate svc
task --> gradle : Complete
deactivate task
@enduml
----

**Notable aspects:**

* Each export entry in the configuration defines its own JQL query, output format, and custom fields.
* The `LegacyJiraProcessor` is invoked first to convert old-style configuration (`iql` property) to the modern `exports` array format.
* Pagination is handled internally by `JiraServerClient` -- callers receive the full issue list.

// Source: core/src/main/groovy/org/docToolchain/tasks/ExportJiraIssuesTask.groovy,
// core/src/main/groovy/org/docToolchain/atlassian/jira/JiraService.groovy

=== Scenario 4: Generate Microsite

Demonstrates jBake-based static site generation.

[plantuml,runtime-site,png]
----
@startuml
actor "Author" as user
participant "Gradle" as gradle
participant "generateSite.gradle" as gen
participant "copyThemes.gradle" as theme
participant "jBake Plugin" as jbake

user -> gradle : ./dtcw local generateSite
gradle -> theme : copyThemes task
activate theme
theme -> theme : Download and extract\nsite theme (ZIP)
theme -> theme : Copy to build/microsite/
theme --> gradle : Theme ready
deactivate theme

gradle -> gen : generateSite task
activate gen
gen -> gen : Copy AsciiDoc sources\nto build/tmp/microsite/
gen -> gen : Fix jBake metadata headers\nin each AsciiDoc file
gen -> gen : Prepare menu structure\nfrom config.microsite.menu
gen -> jbake : jbake.bake()
activate jbake
jbake -> jbake : Process AsciiDoc pages\nApply GSP templates\nGenerate navigation
jbake --> gen : build/microsite/output/
deactivate jbake
gen --> gradle : Site generated
deactivate gen

gradle --> user : Microsite in\nbuild/microsite/output/
@enduml
----

**Notable aspects:**

* Theme management is a separate task (`copyThemes`) that runs before site generation.
* AsciiDoc files require jBake-specific metadata headers (`:jbake-title:`, `:jbake-type:`, etc.) which are automatically fixed during preparation.
* The site uses Groovy Server Pages (GSP) templates from `src/site/templates/`.

// Source: scripts/generateSite.gradle, scripts/copyThemes.gradle
