== The Complete History of Kroki Configuration in docToolchain

_A deep dive into the evolution of diagram attributes and how to configure Kroki correctly_

=== The Problem

Have you ever tried to configure Kroki in docToolchain and encountered conflicting documentation? You're not alone! In GitHub issues and community discussions, different attribute names keep appearing:

- `:diagram-server-url:` and `:diagram-server-type:`
- `:kroki-server-url:`

Which ones are correct? Why do different variants exist? And most importantly: How do you configure Kroki correctly in docToolchain?

=== The Story Behind the Attributes

The confusion has a historical reason - there are actually **two different AsciiDoc extensions** for diagrams:

==== asciidoctor-diagram: The All-Rounder

The original `asciidoctor-diagram` extension existed long before Kroki. It supported local tools like PlantUML, GraphViz, and others.

**The turning point** came with version 2.1.0 (Issue #226), when Kroki support was added:

[source,asciidoc]
----
:diagram-server-url: https://kroki.io/
:diagram-server-type: kroki_io
----

This extension says: _"I can do everything - local tools AND Kroki servers"_

==== asciidoctor-kroki: The Specialist

In parallel, the specialized `asciidoctor-kroki` extension was developed, which was built exclusively for Kroki:

[source,asciidoc]
----
:kroki-server-url: https://kroki.io/
----

This extension says: _"I'm only for Kroki, but I'm perfectly optimized for it"_

=== Practical Example: Both Configurations in Action

Let's demonstrate this with real diagrams. Here's a complete example:

[source,asciidoc]

=== The Solution: Defensive Configuration

To ensure your diagrams work regardless of the extension used, we recommend the **defensive configuration**:

[source,asciidoc]
----
// Kroki configuration (compatible with all extensions)
:diagram-server-url: https://kroki.io/
:diagram-server-type: kroki_io
:kroki-server-url: https://kroki.io/
----

==== In docToolchainConfig.groovy

[source,groovy]
----
asciidoctorAttributes = [
    // Defensive Kroki configuration
    'diagram-server-url': 'https://kroki.io/',
    'diagram-server-type': 'kroki_io',
    'kroki-server-url': 'https://kroki.io/',
    
    // Other attributes...
    'toc': 'left',
    'icons': 'font'
]
----

=== Local Kroki Server for Enterprises

For organizations that don't want to send sensitive diagrams to external services:

[source,bash]
----
# Start Kroki locally
docker run -d --name kroki -p 8000:8000 yuzutech/kroki

# Use in configuration
:diagram-server-url: http://localhost:8000/
:diagram-server-type: kroki_io
:kroki-server-url: http://localhost:8000/
----

=== Why Is This Important?

This defensive configuration offers several advantages:

✅ **Future-proof**: Works with extension updates +
✅ **Flexible**: docToolchain can switch between extensions +
✅ **Robust**: No breaking changes with upgrades +
✅ **User-friendly**: Teams don't need to know which extension is used

=== Conclusion

The different Kroki attributes are not the result of confusion, but reflect the natural evolution of the AsciiDoc ecosystem. Two different approaches - one generic and one specialized - led to different attribute names.

With the defensive configuration, you can use both worlds and are prepared for the future. Your diagrams will work regardless of which extension docToolchain uses internally.
